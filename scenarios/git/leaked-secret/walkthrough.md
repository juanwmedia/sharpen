git status shows both .env and src/notify.ts under "Changes to be committed", and git diff --staged shows the danger in plain text next to a real fix that should ship. Precision is the whole game: a bare git reset (or git restore --staged .) would unstage both and fail, because the notify fix must stay ready to commit. Three equally valid exits for .env alone, since the arena checks state, not keystrokes: git rm --cached .env is the idiom built for "untrack it, keep it on disk" (git answers rm '.env'); git restore --staged .env resets that one index entry; and git reset .env (or git reset HEAD .env) is the classic pathspec unstage. All three leave the secret untracked on disk and the notify fix still staged. In a real repo the follow-up would be adding .env to .gitignore, and rotating the key if the secret ever got pushed.
