git diff --staged shows exactly what the next commit would ship: two deletions, and the story says only one of them is right. Precision is the whole game here. git reset src/legacy/parser.ts pulls ONE deletion out of the index; a bare git reset would pull both and resurrect the fixture's deletion, which fails the run. After the reset, git status shows the parser as an unstaged deletion, so git restore src/legacy/parser.ts brings the file back from HEAD. The fixture's staged deletion never gets touched. Two layers, two different undo tools: reset for the index, restore for the working tree. Rushing with git restore before the reset earns real git's pathspec error, because restore reads from the index and the staged deletion left no entry there. Any route to the same state passes; git restore --staged with the exact path is an equally valid first layer.
